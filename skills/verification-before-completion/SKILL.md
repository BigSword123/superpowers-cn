# 完成前验证

## 概览

未经验证就声明工作完成不是效率，是不诚实。

**核心原则：** 始终先有证据再下断言。

**违反此规则的字面即是违反其精神。**

## 铁律

```
没有全新的验证证据，不得做出完成声明
```

如果你没有在当前消息中运行验证命令，就不能声称它通过了。

## 关卡函数

```
在声明任何状态或表达满意之前：

1. 识别：什么命令能证明此声明？
2. 运行：执行 完整 命令（全新的、完整的）
3. 阅读：完整输出，检查退出码，统计失败数
4. 验证：输出是否确认了声明？
   - 如果否：用证据陈述实际状态
   - 如果是：附带证据陈述声明
5. 只有这时才：做出声明

跳过任何一步 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不足够 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 之前的运行，"应该能通过" |
| Linter 干净 | Linter 输出：0 个错误 | 部分检查，外推 |
| 构建成功 | 构建命令：exit 0 | Linter 通过，日志看起来良好 |
| Bug 已修复 | 测试原始症状：通过 | 代码已更改，假设已修复 |
| 回归测试有效 | 红绿循环已验证 | 测试通过一次 |
| 智能体已完成 | VCS diff 显示变更 | 智能体报告"成功" |
| 需求已满足 | 逐行核对清单 | 测试通过 |

## 红色警示——停止

- 使用"应该"、"可能"、"似乎"
- 在验证之前表达满意（"太好了！"、"完美！"、"完成！" 等）
- 即将在未验证的情况下提交/推送/创建 PR
- 信任智能体的成功报告
- 依赖部分验证
- 想着"就这一次"
- 累了想早点结束
- **任何暗示成功的措辞，但未运行验证**

## 借口预防

| 借口 | 现实 |
|--------|---------|
| "现在应该能工作了" | 运行验证 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "智能体说成功" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查足够了" | 部分证明不了什么 |
| "换个说法规则就不适用了" | 精神大于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该能通过了" / "看起来正确"
```

**回归测试（TDD 红绿循环）：**
```
✅ 编写 → 运行（通过） → 回退修复 → 运行（必须 失败） → 恢复 → 运行（通过）
❌ "我已经写了一个回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到：exit 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**需求：**
```
✅ 重新阅读计划 → 创建核对清单 → 逐项验证 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**智能体委派：**
```
✅ 智能体报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任智能体报告
```

## 为什么这很重要

来自 24 个失败记忆：
- 你的 人类搭档 说 "我不相信你"——信任破裂
- 未定义的函数被发布——会崩溃
- 缺失的需求被发布——功能不完整
- 时间浪费在虚假的完成 → 重定向 → 返工
- 违反："诚实是核心价值。如果你撒谎，你就会被替换。"

## 何时应用

**始终在以下之前：**
- 任何成功/完成声明的变体
- 任何满意的表达
- 任何关于工作状态的积极陈述
- 提交、创建 PR、任务完成
- 进入下一个任务
- 委派给智能体

**规则适用于：**
- 确切的短语
- 释义和同义词
- 暗示成功
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后才声明结果。

这不可妥协。
