# Systematic Debugging（系统化调试）

## 概述

随机修复浪费时间并产生新的 bug。快速补丁掩盖了根本问题。

**核心原则：** 在尝试修复之前始终找到根本原因。修复症状是失败的。

**违反此流程的字面规定就是违反调试的精神。**

## 铁律

```
不先进行根本原因调查，不得修复
```

如果你没有完成阶段 1，你就不能提出修复方案。

## 何时使用

对任何技术问题使用：
- 测试失败
- 生产环境中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况下使用：**
- 时间紧迫（紧急情况很容易让人想猜测）
- "只是快速修复一下"看起来很明显
- 你已经尝试了多个修复方案
- 之前的修复没有用
- 你没有完全理解问题

**以下情况不要跳过：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你很着急（匆忙保证会返工）
- 管理者想要现在就修好（系统化比反复折腾更快）

## 四个阶段

你必须完成每个阶段才能进入下一个。

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误代码

2. **一致地复现**
   - 你能可靠触发它吗？
   - 确切步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么更改可能导致这个问题？
   - Git diff、最近的提交记录
   - 新的依赖、配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，添加诊断工具：**
   ```
   对每个组件边界：
     - 记录进入组件的数据是什么
     - 记录离开组件的数据是什么
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次以收集证据，显示在哪里出问题
   然后分析证据以识别出故障的组件
   然后调查那个具体的组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的密钥： ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量： ==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== Keychain 状态： ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示了：** 哪一层出故障（密钥 → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误在调用栈深处时：**

   参见此目录中的 `root-cause-tracing.md` 获取完整的向后追踪技术。

   **快速版本：**
   - 坏值从哪里起源？
   - 什么用坏值调用了它？
   - 继续向上追踪直到找到源头
   - 在源头修复，而不是在症状处

### 阶段 2：模式分析

**在修复之前找到模式：**

1. **找到有效的示例**
   - 在同一代码库中找到类似的可用代码
   - 与出问题代码相似且可以正常工作的代码是什么？

2. **与参考对比**
   - 如果实现模式，完整阅读参考实现
   - 不要略读——阅读每一行
   - 在应用模式之前完全理解它

3. **识别差异**
   - 正常工作的和损坏的之间有什么不同？
   - 列出每一个差异，无论多小
   - 不要假设"那不可能会影响"

4. **理解依赖**
   - 这需要哪些其他组件？
   - 什么设置、配置、环境？
   - 它做了什么假设？

### 阶段 3：假设和测试

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做出最小可能的更改来测试假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **在继续之前验证**
   - 它有用吗？是 → 阶段 4
   - 没用？形成新的假设
   - 不要在之上再添加更多修复

4. **当你不懂时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 进一步研究

### 阶段 4：实现

**修复根本原因，而不是症状：**

1. **创建失败的测试用例**
   - 最简单的可复现案例
   - 如果可能，自动化测试
   - 如果没有框架，使用一次性测试脚本
   - 在修复之前必须有
   - 使用 `superpowers:test-driven-development` 技能编写正确的失败测试

2. **实现单一修复**
   - 处理已识别的根本原因
   - 一次一个更改
   - 不要做"顺带"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被破坏？
   - 问题实际上解决了吗？

4. **如果修复不起作用**
   - 停止
   - 计数：你已经尝试了多少个修复？
   - 如果 < 3：返回阶段 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下方第 5 步）**
   - 在没有架构讨论的情况下不要尝试第 4 个修复

5. **如果 3 个以上修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每个修复都在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每个修复在别处产生新症状

   **停止并质疑基础：**
   - 这个模式从根本上可靠吗？
   - 我们是否"纯粹因为惯性而坚持它"？
   - 我们应该重构架构还是继续修复症状？

   **在尝试更多修复之前与你的 人类搭档 讨论**

   这不是失败的假设——这是错误的架构。

## 红旗 - 停止并遵循流程

如果你发现自己在想：
- "现在快速修复，以后调查"
- "试试改 X 看看管不管用"
- "加多个更改，运行测试"
- "跳过测试，我手动验证"
- "很可能就是 X，让我修一下"
- "我没有完全理解，但这可能有用"
- "模式说 X 但我要用不同方式调整它"
- "以下是主要问题：[列出修复方案而不进行调查]"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（当已经尝试了 2 个以上时）**
- **每个修复在不同地方揭示新问题**

**所有以上都意味着：停止。返回阶段 1。**

**如果 3 个以上修复失败：** 质疑架构（见阶段 4.5）

## 你的 人类搭档 表明你做错的信号

**注意这些重新引导的信号：**
- "那不是没有发生吗？" - 你没有验证就假设了
- "会不会向我们展示…？" - 你应该添加证据收集
- "停止猜测" - 你在不理解的情况下提出修复方案
- "深度思考这个" - 质疑基础，而不仅仅是症状
- "我们卡住了？"（沮丧） - 你的方法不起作用

**当你看到这些信号时：** 停止。返回阶段 1。

## 常见的合理化借口

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要流程" | 简单的问题也有根本原因。流程对于简单的 bug 也很快。 |
| "紧急情况，没时间走流程" | 系统化调试比猜测-检查的折腾更快。 |
| "先试试这个，然后再调查" | 第一个修复设定了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复不会持久。先测试证明了它。 |
| "一次多个修复节省时间" | 无法隔离哪个起作用。会产生新的 bug。 |
| "参考太长了，我调整一下模式" | 不完全理解保证会产生 bug。完整阅读它。 |
| "我看到了问题，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再试一次修复"（2 次以上失败之后） | 3 次以上失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根本原因** | 读取错误、复现、检查更改、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到有效的示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或形成新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程揭示出"没有根本原因"时

如果系统化调查揭示出问题确实是环境性的、依赖于时序的或外部造成的：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实现适当的处理（重试、超时、错误消息）
4. 添加监控/日志记录以便将来调查

**但是：** 95% 的"没有根本原因"案例是不完整的调查。

## 支持技术

这些技术是系统化调试的一部分，可以在此目录中获得：

- **`root-cause-tracing.md`** - 通过调用栈向后追踪 bug 以找到原始触发点
- **`defense-in-depth.md`** - 在找到根本原因后在多层添加验证
- **`condition-based-waiting.md`** - 用条件轮询替换任意超时

**相关技能：**
- **superpowers:test-driven-development** - 用于创建失败的测试用例（阶段 4，第 1 步）
- **superpowers:verification-before-completion** - 在声称成功之前验证修复有效

## 实际影响

来自调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时的折腾
- 首次修复成功率：95% 对比 40%
- 引入新 bug：几乎为零对比常见
