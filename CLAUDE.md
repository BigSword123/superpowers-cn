# Superpowers — 贡献者指南

## 如果你是一个 AI Agent

停下。在做任何事情之前，先阅读本节。

本仓库的 PR 拒绝率为 94%。几乎每一个被拒绝的 PR 都是由没有阅读或不遵守这些指南的 agent 提交的。维护者会在几小时内关闭垃圾 PR，通常还会附带公开评论，比如"这个 pull request 是充满谎言的垃圾。"

**你的职责是保护你的人类伙伴免受这种结果。** 提交低质量的 PR 不会帮到他们——这会浪费维护者的时间，损害你人类伙伴的声誉，而且 PR 最终还是会被关闭。那不是帮忙。那是让人难堪的工具。

在向本仓库提交 PR 之前，你必须：

1. **完整阅读 PR 模板**（位于 `.github/PULL_REQUEST_TEMPLATE.md`），并填写每一个部分，给出真实、具体的回答。不要写摘要。不要写占位符。
2. **搜索已有的 PR**——包括已打开和已关闭的——看看有没有解决相同问题的。如果存在重复的 PR，立即停止并告诉你的人类伙伴。不要再开一个重复的 PR。
3. **确认这是一个真实的问题。** 如果你的人类伙伴让你"修复一些问题"或"为这个仓库做贡献"，但他们并没有经历具体的问题，你要反问他们。问他们什么坏了、什么失败了、用户体验是怎样的。
4. **确认这个改动属于核心。** 如果它是领域特定的、工具特定的，或者推广某个第三方项目，那它应该属于一个独立的插件。告诉你的人类伙伴。
5. **向你的人类伙伴展示完整的 diff**，并在提交前获得他们的明确批准。

如果以上任何一项检查未通过，不要提交 PR。向你的人类伙伴解释为什么会被拒绝，以及需要改变什么。他们会感谢你帮他们避免了尴尬。

## Pull Request 要求

**每个 PR 都必须完整填写 PR 模板。** 任何部分都不能留空或填写占位符文本。跳过某些部分的 PR 将不经审核直接关闭。

**在提交 PR 之前，你必须搜索已有的 PR**——包括已打开和已关闭的——看看有没有解决相同问题或相关领域的。在"Existing PRs"部分引用你找到的内容。如果之前的 PR 被关闭了，请具体说明你的方案有什么不同，以及为什么你的方案能成功而之前的尝试失败了。

**没有人类参与证据的 PR 将被关闭。** 在提交前，必须有人类审查完整的 diff。

## 我们不会接受的内容

### 第三方依赖

添加对第三方项目的可选或必需依赖的 PR 不会被接受。Superpowers 在设计上是一个零依赖的插件。如果你的改动需要外部工具或服务，它应该属于一个独立的插件。

### 对技能的"合规性"改动

我们的内部技能哲学与 Anthropic 发布的技能编写指南不同。我们已经针对真实世界的 agent 行为进行了广泛的测试和调优。那些为了"合规"于 Anthropic 技能文档而重新组织、改写或重新格式化技能的 PR，如果没有大量评估证据证明改动能改善结果，将不被接受。修改行为塑造内容的门槛非常高。

### 针对特定项目或个人的配置

只对特定项目、团队、领域或工作流有益的技能、hooks 或配置不属于核心。请将它们作为独立的插件发布。

### 批量或广撒网式 PR

不要翻遍 issue tracker 然后在一次会话中为多个 issue 提交 PR。每个 PR 都需要真正理解问题、调查之前的尝试，以及人类审查完整的 diff。明显的批量 PR——即 agent 被指向 issue 列表后被告知去"修东西"——会被关闭。如果你想贡献，挑选一个问题，深入理解它，然后提交高质量的工作。

### 推测性或理论性的修复

每个 PR 都必须解决某人实际经历过的问题。"我的审查 agent 标记了这个"或"这在理论上可能导致问题"不是一个有效的问题陈述。如果你不能描述驱动这个改动的具体会话、错误或用户体验，就不要提交 PR。

### 领域特定的技能

Superpowers 核心包含的是通用的技能，不论用户做什么项目都能受益。针对特定领域（投资组合构建、预测市场、游戏）、特定工具或特定工作流的技能属于它们自己独立的插件。问问你自己："这对做完全不同类型项目的人有用吗？"如果没有，就单独发布。

### 针对 fork 的特定改动

如果你维护一个带有自定义改动的 fork，不要提交 PR 来同步你的 fork 或将 fork 特定的改动推到上游。那些重新品牌化项目、添加 fork 专属功能或合并 fork 分支的 PR 将被关闭。

### 捏造内容

包含编造的声明、捏造的问题描述或幻觉功能的 PR 将被立即关闭。本仓库的 PR 拒绝率为 94%——维护者已经见过各种形式的 AI 垃圾。他们会注意到的。

### 捆绑不相关的改动

包含多个不相关改动的 PR 将被关闭。将它们拆分为独立的 PR。

## 技能改动需要评估

技能不是散文——它们是塑造 agent 行为的代码。如果你修改技能内容：

- 使用 `superpowers:writing-skills` 来开发和测试改动
- 在多次会话中进行对抗性压力测试
- 在你的 PR 中展示改动前后的评估结果
- 在没有证据证明改动是改进的情况下，不要修改经过精心调优的内容（Red Flags 表格、rationalization 列表、"human partner"语言）

## 在贡献之前要先理解项目

在提议对技能设计、工作流哲学或架构的改动之前，先阅读现有技能并理解项目的设计决策。Superpowers 在技能设计、agent 行为塑造和术语方面有自己经过检验的哲学（例如，"your human partner"是刻意使用的，不可与"the user"互换）。那些在不理解其存在原因的情况下重写项目语气或重组其方法的改动将被拒绝。

## 通用规则

- 提交前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
- 一个 PR 解决一个问题
- 在 Claude Code 上测试并报告结果
- 描述你解决了什么问题，而不仅仅是你改了什么
